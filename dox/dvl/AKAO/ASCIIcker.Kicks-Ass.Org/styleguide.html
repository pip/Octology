Differences between XHTML and HTML

Style Guide by Eric Wright, Lead Front-End Developer. Draft 5 October 2005. For more information see the W3C XHTML spec.

For an excellent overview of why web standards are A Good Thing, see Developing With Web Standards.

1. Elements Must Be Properly Nested

In HTML, some elements can be improperly nested within each other, like this: 

<b><i>This text is bold and italic</b></i>

In XHTML, all elements must be properly nested within each other, like this: 

<b><i>This text is bold and italic</i></b>

Note: A common mistake in nested lists is to forget that the inside list must be within an <li> element. This is wrong (missing a </li>): 

<ul>
  <li>Coffee</li>
  <li>Tea
    <ul>
      <li>Black tea</li>
      <li>Green tea</li>
    </ul>
  <li>Milk</li>
</ul>

This is correct (notice that we have inserted a </li> tag after the </ul> tag): 

<ul>
  <li>Coffee</li>
  <li>Tea
    <ul>
      <li>Black tea</li>
      <li>Green tea</li>
    </ul>
  </li>
  <li>Milk</li>
</ul>

 

2. Documents Must Be Well-formed

Nesting

All XHTML elements must be nested within the <html> root element. All elements must be in pairs and correctly nested within their parent element. The basic document structure is: 

<html>
  <head> ... </head>
  <body> ... </body>
</html>

DOCTYPE

All documents should also have a proper DOCTYPE, which should be the very first thing in the document. These are proper XHTML DOCTYPEs (I recommend XHTML Transitional.): 

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN"
"http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN"
"http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Frameset//EN"
"http://www.w3.org/TR/xhtml1/DTD/xhtml1-frameset.dtd">

Character Encoding

All XHTML documents must specify their character encoding, as soon as possible. The most common way is to include this information as your first meta tag.

There are several options for character encoding, but I recomment UTF-8. 

<meta http-equiv="content-type" content="text/html; charset=UTF-8" />

3. Tag Names Must Be Lower Case

This is because XHTML documents are XML applications, and XML is case-sensitive. This means <br> and <BR> are two different animals.

This is wrong: 

<BODY>
  <P>This is a paragraph.</P>
</BODY>

This is correct: 

<body>
  <p>This is a paragraph.</p>
</body>

4. All Elements Must Be Closed

Non-empty elements must have an end tag.

This is wrong: 

<p>This is a paragraph
<p>This is another paragraph

This is correct: 

<p>This is a paragraph</p>
<p>This is another paragraph</p>

Empty Elements Must Also Be Closed

Empty elements must either have an end tag or the start tag must end with />.

This is wrong: 

This is a break<br>
Here comes a horizontal rule:<hr>
Here's an image <img src="happy.gif" alt="Happy face">

This is correct: 

This is a break<br />
Here comes a horizontal rule:<hr />
Here's an image <img src="happy.gif" alt="Happy face" />

Compatibility Note: To make XHTML compatible with today's browsers, add an extra space before the "/" symbol like this: 

<br />
<hr />

5. Attributes Must Be Well-Formed

Attributes Must Always Be Quoted

This is wrong: 

<p id=summary>Not much happened.</p>

This is correct: 

<p id="summary">Not much happened.</p>

Attributes Must Always Have a Value

In HTML, attributes can exists without a value. 

<ul compact>
<option selected>
<td nowrap> text </td>
<input type="radio" checked>
<input type="checkbox" checked>

This isn't allowed in XHTML. Instead, write attributes without a value as attribute="attribute", like this: 

<ul compact="compact">
<option selected="selected">
<td nowrap="nowrap"> text </td>
<input type="radio" checked="checked" />
<input type="checkbox" checked="checked" />

Organizing CSS

There are several possible ways to organize CSS, and it's hard to say which one is "best". Below is the method I use.

0. Imported Styles Before Anything Else

Any imported stylesheets must be listed first. We split all browser-specific rules into their own stylesheets, which are imported at the top of main.css.

1. List the "Basics" First

All CSS applied to straight tags comes first. This includes html, body, p, hn, ul, ol, li, :link, :visited, :hover, :active, table, form, input

2. List Elements As Ordered In The Markup

Next, divide elements into sections based on the order they occur the in markup. For example: 

/* #header
--------------------------------------------------*/
div#header { }
div#header h1 { }


/* #page_content
--------------------------------------------------*/
div#page_content { }
div#page_content p { }


/* #nav_menu
--------------------------------------------------*/
div#nav_menu { }
div#nav_menu li { }
div#nav_menu li.current { }


/* #footer
--------------------------------------------------*/
div#footer { }
div#footer #copyright { }

I also use a row of 50 dashes as a header underline. This make it easier to find sections when scrolling through the document. Also, you’ll be able to search for a certain section using /* #section_id.

3. Organize Page-Specific Elements By Page

Finally, have sections for elements that apply only to certain pages. For example: 

/* 2.0 Settings
--------------------------------------------------*/
div#user_settings { }
input#save_user { }


/* 4.0 Personnel
--------------------------------------------------*/
div#personnel { }
div#personnel h3.first_name { }

4. Organize Attributes Into Three Logical Sections

I organize the attributes inside a CSS declaration into three logical sections. The basic prototype is this: 

div#whatever {
    /* Behavioral attributes */
    position: ;
    display: ;
    top: ;
    left ;
    height ;
    width: ;
    float ;

    /* The Three Musketeers */
    border: ;
    margin: ;
    padding: ;

    /* Look and Feel */
    background: ;
    color: ;
    font: ;
    text-decoration: ;
    text-indent: ;
    list-style: ;
}

Section 1: Behavioral Attributes

This section sometimes includes only position and display, but if necessary it contains all attributes that determine the way this element behaves in relation to other elements and the flow of the document (with the noted exception of the Three Musketeers in Section 2).

Section 2: The Three Musketeers

This section is easy: it's always border, margin, and padding. Occasionally something related like border-collapse might make sense here, but that’s rare.

Section 3: Look and Feel

Finally, the third section is where you find all cosmetic attributes. Often this is only background and color, but it also contains everything to do with text (like font), as well as other presentation attributes like list-style.
